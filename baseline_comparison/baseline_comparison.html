Baseline Comparison

A step-by-step technical breakdown of three deterministic literature-retrieval pipelines

◉ Dynamic-LR  (this repo) ◉ PRS  cherryann518/paper-retrieval-system ◉ PF  WoosungLim01/Paper-Feed

Shared Pipeline Skeleton

Survey Config→ Query Generation→ API Fetch→ Normalize→ Identity Resolution→ Deduplication→ Relevance Scoring→ Filtering→ Persist→ Publish→ Loop Control

All three systems follow roughly this order. The differences lie entirely in how each step is implemented — depth, strategy, and engineering choices that compound across a multi-run pipeline.

Literature Grounding

Three recent papers on AI-assisted systematic review provide the research context for Dynamic-LR's design. Their key findings constrain how each pipeline step should be implemented — and where the three systems diverge from best practice.

InsightAgent

Qiu et al., 2025 · arXiv:2504.14822

Demonstrates that systematic review benefits from a provenance graph: every high-level claim links back to paper-level evidence. Semantic partitioning before parallel work and isolated per-subdomain memory reduce conflation errors. Human overrides must be stored as new decision events, never silently overwriting prior choices.

Agentic AutoSurvey

Liu et al., 2025 · arXiv:2509.18661

Shows that flat, single-pass synthesis degrades on large corpora — hierarchical aggregation is needed. Establishes the value of multi-source retrieval, explicit caching with TTL, exponential backoff with jitter, and a 12-dimensional quality rubric. Warns against title-similarity-only deduplication and self-reported quality as the sole evaluator.

LatteReview

Rouzrokh et al., 2025 · arXiv:2501.05468

Introduces a round-based workflow schema: metadata filter → title/abstract screen → conflict resolution → evidence extraction. Provider abstraction, typed inputs/outputs, conditional escalation, and async batch execution are its most transferable ideas. Cautions that results vary by task and inclusion prevalence — a high AUC does not eliminate low precision in rare-inclusion settings.

Three-Layer Architecture Principle

All three papers converge on a common structural principle — keep these three layers strictly separate:

Acquisition Curation Presentation
Acquisition

Source queries, raw API payloads, rate limiting, caching, normalization, identifier resolution, deduplication. No relevance judgment here.

Curation

Relevance scoring, inclusion/exclusion decisions, topic clusters, evidence extraction, contradiction flags, user overrides. No raw API calls here.

Presentation

Static site data, corpus views, exports, survey draft, audit reports. No artifact may be the sole source of truth. The database and run artifacts are authoritative.

Canonical Data Contracts

The literature mandates five structured record types that all architectures should write consistently:

Evaluation Framework

For fair baseline-vs-agent comparison the literature recommends two complementary modes:

Frozen-Corpus Comparison

Build one normalized, deduplicated corpus snapshot, then feed the same snapshot and acceptance criteria to all architectures. This isolates screening, organization, and synthesis decisions from retrieval variance.

End-to-End Retrieval Comparison

Run each architecture from the same config and time window. Record: source coverage, query count, unique candidates, deduplication behavior, accepted records, runtime, API calls, cache hits, retries, failures — and target-paper recall when ground truth is available.

Research Principle: Do not compare only prose quality. A system that writes a good-looking survey but cannot reproduce source inclusion, citations, and run state is not a credible research pipeline. Dynamic-LR's structured run artifacts and SQLite persistence are prerequisites for this kind of comparison — not optional extras. — Derived from Agentic AutoSurvey §5, LatteReview §4.2, InsightAgent §6

STEP 1 Survey Config Loading

All three systems read from a survey_config.json file. The design of the config dataclass determines what the rest of the pipeline can use.

Dynamic-LR

Config is stored in the shared app/state/schemas.py as a Pydantic/dataclass SurveyConfig. Reads topic_overview, research_questions, question_context, query_hints, timeline_from_year, timeline_to_year, min_relevance_score, and an optional baseline dict for architecture-specific settings like max_queries, max_results_per_query, enable_sqlite. Unknown fields are silently ignored — the config is forward-compatible.

PRS

SurveyConfig dataclass in src/fetch.py. Loads topic_overview, timeline_from_year, timeline_to_year, fields_of_study, min_relevance_score, max_candidates_per_run, sources, and rank_method. Research questions and hints are intentionally not part of the schema — the system relies on a single topic string and uses the LLM to refine it. Config can also be overridden at CLI via --topic.

PF

SurveyConfig dataclass in app/config.py. Loads topic_overview, research_questions, question_context, query_hints, timeline_from_year, timeline_to_year, min_relevance_score, and a search_keywords field (falls back to query_hints if absent). CLI supports --topic, --days-back, and --days-buffer overrides. Config is strict — missing required fields will throw a KeyError.

Key Differences

Architecture-specific config block

Only Dynamic-LR supports an optional "baseline": { ... } sub-object in the config JSON. This means baseline-specific settings (max queries, use bulk search, SQLite path) live next to survey settings without polluting shared schema. PRS and PF have no such separation — all settings are flat and global.

Pros — Dynamic-LR approach
  • Architecture-specific tuning without breaking other architectures
  • Config file documents exactly what the baseline pipeline uses
  • Safe defaults mean the block can be omitted entirely
Cons
  • Nested config is slightly harder to read and document
  • Requires the pipeline to safely navigate optional nested keys

Research questions in config

Dynamic-LR and PF both include research_questions as first-class config fields and use them downstream (for scoring or query generation). PRS ignores them — the intent is that the LLM should derive diverse queries from a single topic string rather than having the user enumerate them.

Pros — including research questions
  • Each question maps directly to a targeted API query
  • Scoring query text is richer — abstracts are compared against the actual research intent
  • Makes the retrieval intent explicit and auditable
Cons
  • User must write good questions; poor questions degrade query quality
  • Long questions perform poorly as verbatim search queries (addressed by query cleaning)

STEP 2 Query Generation

The quality and diversity of queries is the single most controllable lever for retrieval coverage. Different strategies make fundamentally different trade-offs between diversity, determinism, and relevance signal.

Dynamic-LR — query_builder.py

Generates up to max_queries from four sources: (1) topic_overview, (2) each research_question, (3) each query_hint, (4) a combined query built from the top-N non-stopword tokens across all config fields ranked by frequency, with ties broken alphabetically for reproducibility. All queries are cleaned (control chars, whitespace collapse, length cap) and deduplicated case-insensitively with first-occurrence order preserved.

PRS — src/agent.py

Starts with a single topic string. An ML acronym expansion table resolves abbreviations before the first search (e.g. rag → retrieval augmented generation). After scoring, if results are below threshold, an Ollama LLM generates a revised query based on why results failed. This can repeat up to MAX_REFINEMENT_ROUNDS=2 times. Without Ollama, the search is a single fixed query.

PF — app/fetcher.py

dict.fromkeys 保留插入顺序进行去重。无组合查询，无主题扩展。最小预处理：通过正则表达式在发送前去除标点。">Uses only search_keywords (or query_hints as fallback). Research questions are explicitly excluded — the comment notes they are "natural language, not suitable for API search." Each keyword becomes one query. Deduplication preserves insertion order via dict.fromkeys. No combined query. No topic expansion. Minimal pre-processing: punctuation stripped via regex before sending.

Deep Dive

Dynamic-LR — The Combined Query

The combined query is the most distinctive element. Rather than sending the full topic string, it extracts the most informationally dense tokens across all config fields:

# Sort by (-count, token) — stable, deterministic
ranked = sorted(counter.items(), key=lambda kv: (-kv[1], kv[0]))
top = [tok for tok, _ in ranked[:max_tokens]]
combined = " ".join(top)
Pros
  • Compact, high-density query — avoids token budget waste on stopwords
  • Cross-field signal: terms that appear in both topic and questions get boosted
  • Fully deterministic — same config always yields the same combined query
Cons
  • Loses word order and phrasing — keyword soup may match unrelated papers
  • Short abstracts may not contain individual tokens even when highly relevant
  • Top-frequency tokens may be obvious terms (e.g. "learning") that add noise

PRS — LLM-Driven Refinement Loop

PRS's Ollama call produces a new search string after examining why the current batch scored poorly. Four failure modes trigger different prompt hints:

REFINEMENT_HINTS = {
    "no_papers":          "Broaden the query...",
    "top_score_low":      "Rephrase using precise technical terms...",
    "too_few_good_papers":"Try synonyms, abbreviations...",
    "flat_scores":        "Narrow the query to a specific method...",
}
Pros
  • Adaptive — poor first results don't end the run
  • Can recover from under-specified or over-broad initial queries
  • Produces domain-specific synonyms a static list would miss
Cons
  • Non-deterministic — two identical runs may produce different queries and results
  • Requires a running Ollama instance; unavailability silently falls back to no refinement
  • LLM may drift to adjacent domains (e.g. "RAG" → biology instead of retrieval)
  • Violates baseline rules for this project

PF — Keyword-Only Strategy

Pros
  • Short, clean queries perform well on metadata-search APIs like S2
  • No question-to-query translation required
  • Simple and predictable
Cons
  • Research questions (which often contain the richest intent signal) are unused
  • No combined query means important cross-field term combinations are never tried
  • Coverage is bounded by however many explicit keywords the user wrote
Insight: Dynamic-LR generates 3–12 queries from a single config automatically. PF generates only as many as explicit keywords. PRS generates 1–3 but adapts them. For a fixed-config automated pipeline, Dynamic-LR's strategy provides the best coverage-per-config-field ratio without any LLM dependency.
Research Principle — Retrieval is recall-oriented: LatteReview explicitly states that database filters and careful queries should handle conditions that do not require AI. The baseline must log every query with a query_origin tag (config|deterministic_template|agent) so that retrieval coverage and query effectiveness are measurable across architecture comparison runs. — LatteReview §3.1; Agentic AutoSurvey §3.2; Dynamic-LR agent context §4.1

The fetch step determines how many and how diverse the candidate papers are. Source breadth, pagination strategy, and API client robustness all compound here.

Deep Dive

PRS — Exponential Backoff with Jitter

The most sophisticated 429 handling of the three. It reads the HTTP Retry-After header when present, otherwise falls back to exponential backoff with a random multiplier (1.0–1.5×) to spread retries:

wait = min(INITIAL_BACKOFF * (2**attempt), 60.0)
wait *= random.uniform(1.0, 1.5)   # jitter prevents thundering herd
Pros
  • Respects the server's stated retry window
  • Jitter prevents multiple concurrent processes from all retrying at the same moment
  • Bounded max wait (60 s) avoids infinite hangs
Cons
  • Only implemented for 429; 5xx errors raise immediately

PF — Async Concurrent Multi-Source Fetch

asyncio.gather(
    fetch_arxiv(queries, days_back),
    fetch_semantic_scholar(queries, days_back),
    fetch_openalex(queries, days_back),
    return_exceptions=True
)

All three sources run in parallel. If one raises, it returns an exception and the others continue — true fail-soft fetch. Total wall-clock time is the max of the three source times, not the sum.

Pros
  • 2–3× faster than sequential fetches for multi-source pipelines
  • Source failures are isolated — one failure does not abort the run
  • OpenAlex is free and keyless — no rate-limit quota to share with S2
Cons
  • Requires asyncio throughout the call stack
  • Harder to reason about request ordering and rate limits across sources
  • Fixed 60 s retry wait ignores the Retry-After header

PF — arXiv Date-Aware Pagination

PF stops paginating a query as soon as it encounters an entry older than the date window — assuming descending date order. This is efficient but brittle:

Pros
  • Never fetches more pages than needed for the time window
  • Automatically caps at 300 results per query
Cons
  • Assumes results are strictly sorted by date — not guaranteed for all queries
  • A single mis-dated entry can prematurely terminate pagination, missing valid recent papers
Insight: Dynamic-LR has the most robust retry logic (5xx + TTL cache). PRS has the best backoff strategy. PF has the best source breadth and the only async implementation. The ultimate baseline needs PRS's backoff inside PF's async multi-source fetch, cached in Dynamic-LR's SQLite api_cache.
Research Principle — Cache, retry, resume: Agentic AutoSurvey establishes that production retrieval pipelines must: cache raw API responses with an explicit TTL and request fingerprint; use exponential backoff with jitter on rate limits; allow safe fallback to cached results or an alternate source; checkpoint after each durable stage; and distinguish partial_success, retryable_failure, permanent_failure, and skipped_by_policy. Every query-level failure must be observable — never silently dropped. — Agentic AutoSurvey §4.3; Dynamic-LR agent context §4.3

STEP 4 Normalization

Normalization converts heterogeneous API responses into a shared candidate schema. The quality and completeness of this step determines what the deduplicator and scorer can use.

Dynamic-LR — normalizer.py

Converts S2 API dicts into PaperCandidate objects (Pydantic/dataclass). Extracts paperId, corpusId, title, abstract, year, publicationDate, authors, venue, citationCount, referenceCount, influentialCitationCount, fieldsOfStudy, publicationTypes, openAccessPdf, and all externalIds (DOI, arXiv, PMID, ACL, MAG). Stores the rich identifier set in a nested PaperIdentifiers sub-object for use by the identity resolver.

PRS — src/normalize.py

Separate normalizers for each source: normalize_s2 and normalize_arxiv. Both return a PaperRecord dataclass with paper_id, title, authors, year, abstract, doi, arxiv_id, s2_paper_id, pdf_url, venue, citation_count, sources, and source_queries. Also includes a merge_paper_records function that deduplicates a list of records by paper_id during normalization itself.

PF — app/fetcher.py

Normalization is embedded directly in the fetcher module as _normalize_arxiv, _normalize_s2, and _normalize_openalex. Each fills a canonical candidate dict from _empty_candidate(). Notable: OpenAlex stores abstracts as an inverted index — PF includes a _reconstruct_abstract function that rebuilds the text from word-position mappings. Stores source as a string and appends to a source_hits list for cross-source tracking.

Key Differences

OpenAlex abstract reconstruction (PF only)

OpenAlex does not return abstracts as plain text. It uses an inverted index format where each word maps to a list of positions. PF reconstructs the original text from this:

def _reconstruct_abstract(inverted_index: dict) -> str:
    positions = {}
    for word, pos_list in inverted_index.items():
        for pos in pos_list:
            positions[pos] = word
    return " ".join(positions[i] for i in sorted(positions))
Pros
  • Enables scoring from OpenAlex abstracts without a second API call
  • Handles papers with no conventional abstract field
Cons
  • Some OA records have no inverted index (abstract = empty string)
  • Reconstruction is exact but loses original whitespace/punctuation style

STEP 5 Identity Resolution

Stable identity is the foundation of deduplication. A paper retrieved from three sources with slightly different metadata must resolve to the same canonical record every time.

Identity Priority Chains

Dynamic-LR

1. DOI (normalized)
2. arXiv ID (version stripped)
3. Semantic Scholar paperId
4. S2 corpusId
5. PMID / ACL / MAG
6. SHA-256(title_norm|year|first_author)

PRS

1. DOI (lowercased)
2. arXiv ID (version stripped)
3. Semantic Scholar paperId
4. SHA-256(title_norm|year)

PF

1. DOI (URL-prefix stripped)
2. arXiv ID (URL-prefix stripped)
3. OpenAlex ID (path segment)
4. S2 paperId
5. MD5(title_norm|year|first_author_last)
Normalization Rules

DOI Normalization

All three strip URL prefixes (https://doi.org/, doi:) and lowercase. Dynamic-LR additionally strips whitespace.

Why it matters
  • S2 returns 10.1234/abc; arXiv may return https://doi.org/10.1234/abc
  • Without normalization, identical papers get two IDs and are never merged

arXiv ID Normalization

All three strip the version suffix (v1, v2 etc.). S2 and PRS also strip arxiv: prefix. PF strips the full URL prefix.

Why it matters
  • 2301.00001v2 and 2301.00001 are the same paper
  • S2 often returns ArXiv (capital A) while arXiv API returns lowercase

Title Fingerprint Fallback

Used only when no stable identifier is present. Dynamic-LR and PRS use SHA-256; PF uses MD5. Dynamic-LR adds first-author first token; PF adds first-author last name; PRS uses title+year only.

Risk
  • Title varies across sources (subtitle, case, punctuation)
  • Two records for the same paper with slightly different titles get different fingerprints
  • Including author reduces false positives but requires consistent author format
Warning: PF uses MD5 for fingerprinting. While MD5 collision probability is negligible at these scales, SHA-256 is a better habit for new code. More importantly, PF's fingerprint does not normalize for Unicode — an author name with an accented character in one source vs. ASCII in another will produce different fingerprints for the same paper.
Research Principle — Conservative deduplication: Agentic AutoSurvey uses 90% title similarity as a merge rule. The agent context document explicitly warns against this: short, generic, or near-identical titles will collapse distinct papers. The correct order is exact identifiers first, then exact normalized title + strong author/year agreement, then fuzzy candidates labeled possible_duplicate for inspection rather than auto-merged. A false merge is worse than a retained duplicate because it silently destroys coverage and provenance. — Dynamic-LR agent context §4.2; Agentic AutoSurvey §3.3

STEP 6 Deduplication & Merge

Deduplication has two phases: collapsing duplicates within a single run (intra-run), and matching candidates against the persistent database (cross-session). The merge strategy determines whether the final record is the most complete or just the first seen.

Dynamic-LR — deduper.py

Intra-run: dedupe_candidates groups candidates by stable_paper_id and merges them via explicit preference rules: keep longer title, keep longer abstract, fill missing year, keep richer author list, fill missing URLs, take max citation count.

Cross-session: resolve_against_db queries SQLite for a matching paper using DOI, arXiv, S2 ID, and fingerprint. If found, merges the candidate into the existing record and records a MergeEvent.

PRS — src/store.py + normalize.py

Intra-run: merge_paper_records in normalize.py deduplicates by paper_id immediately after normalization, merging sources and queries, keeping first-seen for content fields (title, abstract, year).

Cross-session: upsert_papers_batch in SQLite checks for an existing paper and calls _merge_record, again preserving the existing content fields and only filling what is missing. Batched in chunks of 50 with transactions.

PF — app/deduplicator.py

Intra-run only: deduplicate iterates all candidates, groups by identity key, and when a collision is found it computes a _completeness_score (count of non-empty fields) for both records. The higher-scoring record becomes the base; missing IDs are filled from the other.

No cross-session SQLite. Cross-run dedup relies on checking paper_id against existing NDJSON IDs during publishing — a full file scan each run.

Deep Dive — Merge Strategies

PF's Completeness-Based Merge Base Selection

def _completeness_score(candidate: dict) -> int:
    fields = ["title", "abstract", "authors", "year", "url",
              "doi", "arxiv_id", "openalex_id", "semantic_scholar_id", "pdf_url"]
    return sum(1 for f in fields if candidate.get(f) not in (None, "", []))
Pros
  • Maximizes metadata completeness in the merged record
  • If OpenAlex has an abstract but S2 does not, the OA record becomes the base
  • Simple and transparent — no field-by-field judgment calls
Cons
  • Completeness score weights all fields equally — a record with a wrong short title scores the same as one with a correct title and no abstract
  • Can demote a more authoritative source (S2 with DOI) in favor of a more complete one (OA with abstract but no DOI)

Dynamic-LR's Explicit Merge Rules

# Title: keep longer non-empty
if new.title and len(new.title) > len(primary.title or ""):
    primary.title = new.title
# Abstract: keep longer non-empty
if new.abstract and len(new.abstract) > len(primary.abstract or ""):
    primary.abstract = new.abstract
# Citation count: take max (latest observed)
setattr(primary, attr, max(cur, new_val))
Pros
  • Each field has a documented, justified preference rule
  • Never overwrites a good value with null
  • Citation counts are always the latest observed (max), not the first seen
Cons
  • Longer title is not always better — an erroneous long title will displace a correct short one
  • Rules need to be maintained as the schema evolves
Insight: The ideal merge strategy combines Dynamic-LR's explicit per-field rules with PF's completeness-based base selection: pick the richer record as the starting point, then fill individual missing fields from the other using the explicit preference rules. Neither system does this today.

STEP 7 Relevance Scoring

Scoring is the step that most directly determines which papers are accepted. The choice between lexical and semantic scoring is the most consequential architectural decision in this pipeline.

Dynamic-LR — scorer.py

Lexical-only weighted formula. No ML dependencies. All components are normalized to [0,1] and stored on the candidate for auditability.

PRS — src/rank.py

SBERT (all-MiniLM-L6-v2) as primary scorer; TF-IDF cosine and recency as companions. Primary method is configurable (sbert / tfidf / recency). Uses Python object id(paper) as the temporary key to avoid copying papers.

PF — app/scorer.py + scorer_sbert.py

SBERT + FAISS as primary; TF-IDF as companion. Both scores stored on every paper. SBERT uses a persistent embedding cache and FAISS IndexFlatIP with L2-normalized vectors (cosine similarity by inner product). GPU/MPS auto-detected.

Deep Dive — Scoring Formulas

Dynamic-LR — Lexical Weighted Formula

score = 0.35 × title_keyword_overlap
+ 0.30 × abstract_keyword_overlap
+ 0.15 × query_phrase_match
+ 0.10 × recency_score
+ 0.05 × identifier_score
+ 0.05 × citation_score

Each component is a ratio bounded to [0,1]:

Keyword overlap

Jaccard-style intersection: |config_terms ∩ paper_tokens| / |config_terms|. Stopwords are removed from both sets. Config terms are the union of all non-stop tokens from topic, questions, and hints.

Phrase match

Returns 1.0 if any generated query appears verbatim (lowercased) in the title+abstract; 0.5 if at least 50% of the phrase tokens appear; 0.0 otherwise.

Recency score

1.0 inside the configured year window; linear decay at 0.1/year outside; 0.5 if year is missing (partial credit).

Pros — Lexical
  • Fully deterministic — no model, no GPU, no internet
  • Every score component has a human-readable explanation
  • Identifier and citation components reward metadata quality
  • Score is consistent across languages if token overlap is high
Cons — Lexical
  • Misses semantically related papers that don't share surface tokens (e.g. "neural network" vs "deep learning")
  • Query-phrase match rewards papers that happen to quote the query rather than understand it
  • Weights are hand-tuned — no empirical calibration

PRS / PF — SBERT Semantic Scoring

# PRS: rank_sbert
query_embedding = model.encode(query, convert_to_tensor=True)
paper_embeddings = model.encode([_paper_text(p) for p in papers])
scores = cos_sim(query_embedding, paper_embeddings)[0].tolist()

# PF: FAISS IndexFlatIP (equivalent but faster)
faiss.normalize_L2(normalized)          # unit vectors
index.add(normalized)                   # add paper vectors
faiss.normalize_L2(query_emb)
scores, indices = index.search(query_emb, k=len(papers))
Pros — SBERT
  • Captures semantic relationships: synonyms, paraphrases, related methods
  • Single score without hand-tuned weights
  • State-of-the-art recall on broad research topics
Cons — SBERT
  • ~80 MB model download on first run; slow on CPU without GPU
  • Score is not directly interpretable — no component breakdown
  • Score range varies by query — a threshold of 0.3 means different things for different topics
  • Sensitive to the quality of the query text passed to the encoder

PF — FAISS vs. PRS — cos_sim

PRS uses sentence_transformers.util.cos_sim (PyTorch tensors). PF uses FAISS IndexFlatIP with L2-normalized vectors — inner product of unit vectors equals cosine similarity but FAISS is 10–100× faster on large candidate sets.

PF FAISS advantage
  • Scales to thousands of candidates without per-call slowdown
  • L2 normalization step is explicit and readable
  • Embedding cache means only new papers are encoded on repeat runs
FAISS trade-offs
  • Adds a native binary dependency; harder to install in restricted environments
  • IndexFlatIP is exact (100% recall) but brute-force — not an ANN index

PF — Dual-Score Storage

PF runs both TF-IDF and SBERT on every paper and stores both. The primary filter uses sbert_score, but tfidf_score is persisted for later inspection. The publisher can also retro-score existing papers that were saved before dual scoring was added.

Pros
  • Enables post-hoc analysis: which papers score high on lexical but low on semantic (or vice versa)?
  • Backward compatible — old records can be upgraded in-place
Cons
  • Running both scorers doubles compute time
  • Only one is used for the accept/reject decision — the other is decorative unless consumed downstream
Insight: The ultimate baseline should use SBERT as the primary signal and lexical scores as the explainability layer. A paper that scores high on SBERT but low on keyword overlap is worth investigating; a paper that scores low on both is a confident reject. PF's dual-output pattern is the right design — adopt it, but use Dynamic-LR's lexical formula (not just TF-IDF) as the companion because it includes recency, citation, and identifier signals that pure TF-IDF misses.
Research Principle — ScreeningDecision record: LatteReview's round-based schema maps directly to how scoring results should be persisted. Every accept/reject decision should be stored as a ScreeningDecision with: the pipeline stage (metadata_filter / baseline_score / agent_review / human_override), the decision itself, the score, a criteria version string, a concise rationale (not hidden chain-of-thought), and the supporting fields that drove the decision. This makes the baseline and agent outcomes structurally comparable — not just numerically comparable. — LatteReview §3.2; Dynamic-LR agent context §3.3

STEP 8 Filtering & Accept / Reject

Filtering is the gate between candidate pool and accepted database. The quality of reject records determines how well the system can be tuned and audited over time.

Dynamic-LR — filters.py

Applies a sequence of deterministic filter checks. Every rejected candidate receives a typed reason code (missing_title, below_relevance_threshold, outside_timeline, duplicate_merged, malformed_metadata, api_error, target_not_found, other) and a structured evidence dict containing the threshold, matched terms, and missing terms.

PRS — src/agent.py

Filtering is embedded in the agent loop via results_acceptable. Checks three conditions simultaneously: top_score >= SCORE_TOP_MIN, good_count >= MIN_GOOD_PAPERS, and top - median_top5 >= MIN_SCORE_GAP. If the batch fails, it triggers a new query rather than saving rejects. Individual paper rejects are written to SQLite with a reason string.

PF — app/run.py

Inline filter in the main pipeline loop. Three conditions: not title, not url (missing metadata), score < min_relevance_score, and year outside timeline. Rejected papers get a reject_reason string field. No separate filter module — all logic lives in run.py.

Deep Dive

PRS — Batch Acceptance Gate

def results_acceptable(ranked):
    scores = [p["relevance_score"] for p in ranked]
    top = scores[0]
    good_count = sum(1 for s in scores if s >= SCORE_GOOD)     # 0.40
    median_top5 = sorted(scores[:5])[len(scores[:5])//2]
    if top < SCORE_TOP_MIN:  return False, "top_score_low"
    if good_count < MIN_GOOD_PAPERS:  return False, "too_few_good_papers"
    if top - median_top5 < MIN_SCORE_GAP:  return False, "flat_scores"
    return True, "ok"
Pros
  • Catches cases where every paper passes a threshold but the whole batch is mediocre
  • "Flat scores" check detects ambiguous queries where all papers score similarly
  • Provides a named failure reason that feeds back into query refinement
Cons
  • Hardcoded thresholds (SCORE_GOOD=0.40, MIN_GOOD_PAPERS=3, MIN_SCORE_GAP=0.08) are not configurable from the survey config
  • Designed to trigger LLM re-query; in a deterministic pipeline it becomes a metric rather than an action

Dynamic-LR — Typed Reject Evidence

{
  "reason": "below_relevance_threshold",
  "score": 0.18,
  "evidence": {
    "threshold": 0.30,
    "matched_terms": ["explainability", "neural"],
    "missing_terms": ["uncertainty", "calibration", "interpretable"]
  }
}
Pros
  • A reviewer can look at any rejected paper and understand exactly why
  • Missing terms reveal which config vocabulary is not being matched — useful for tuning query hints
  • Enables aggregate analysis: which terms are most frequently missing?
Cons
  • Adds storage overhead for every rejected paper
  • Evidence is only meaningful for lexical scoring; SBERT rejects have no analogous explanation
Note on PF's URL requirement: PF rejects papers with no URL regardless of score. OpenAlex records sometimes lack a primary_location.landing_page_url (e.g. preprints without a journal page). This filter may silently drop valid papers that have DOIs but no webpage — worth relaxing to: require URL or DOI.
Research Principle — Cheap source-native filters first: LatteReview's model maps well to this step: apply cheap, rule-based metadata filters in Round 0 before any scoring. Year range, language, document type, open-access flag, and missing-title rules should all be separate named filter conditions, each producing a typed reject reason. This is the "metadata filter" stage of the ScreeningDecision contract — it must be distinguishable from the "baseline_score" stage in audit logs. — LatteReview §3.1; Dynamic-LR agent context §5.1

STEP 9 Persistence & Storage

Persistence determines how well the pipeline maintains state across runs. A system without cross-session state cannot detect duplicates introduced by previous runs, cannot track query performance, and cannot support target-paper detection.

Dynamic-LR — db.py

SQLite as the primary source of truth. Schema has 7 tables: papers, source_hits, rejects, merge_events, runs, api_cache, query_state. All writes use transactions. Export files are written atomically (tmp + rename). api_cache stores raw API responses with a TTL, enabling the pipeline to skip redundant calls on repeat runs.

PRS — src/store.py

SQLite with 4 tables: papers, source_hits, merge_events, rejects. Batched upsert in chunks of 50 with rollback on error. File cache for API responses (one JSON file per request hash). No atomic writes on export. No run history or query state tables.

PF — app/store.py

NDJSON as primary storage. No SQLite. Cross-run duplicate detection reads all existing paper IDs from papers.ndjson on every publish call. Embedding cache stored as data/embeddings_cache.npy (numpy) — the only form of cross-run state. No run artifacts, no API response cache, no query state.

  • Loop control can demote queries that consistently produce zero accepted papers
  • Reveals which query types produce the highest duplicate rate over time
  • Provides evidence for adding or removing query hints from the config

  • Adds a write on every query completion — minor overhead
  • Normalization of query strings must be consistent across runs to match correctly

STEP 10 Publishing & Site Export

Publishing converts internal pipeline state into the shared output files consumed by the static site and other architectures.

Dynamic-LR — publisher.py

Exports from SQLite to: data/papers.ndjson, data/rejects.ndjson, data/run_history.ndjson, data/changelog.md, data/system_status.json. Mirrors to site/data/*.json. Uses atomic writes (write to .tmp, rename) to prevent partial reads. Exports are sorted deterministically before writing.

PRS — No publisher

PRS has no static publishing pipeline. The SQLite database is served directly by a Flask application (src/app.py). This means there is no NDJSON compatibility with the shared project files. Run artifacts are saved under outputs/runs/{run_id}/ as JSON files.

PF — app/publisher.py

Reads all existing NDJSON on every publish, checks for new papers by paper_id, appends new ones, and writes the full collection to site/data/papers.json. Also retro-scores existing papers missing tfidf_score or sbert_score. Copies config and changelog to site/data/. Writes score fields as null for legacy papers without those fields.

Key Differences

Atomic writes (Dynamic-LR only)

write_data  -> papers.ndjson.tmp
os.rename("papers.ndjson.tmp", "papers.ndjson")
Why it matters
  • If the pipeline crashes mid-write, the published file is not left in a partial state
  • A reader that opens the file while a write is in progress sees the old complete file, not garbage
Cost
  • Requires the tmp file and final file to be on the same filesystem (same partition)
  • Adds a stat + rename syscall per export file

PF — Retro-scoring on publish

When PF's publisher detects existing papers in papers.ndjson that are missing tfidf_score or sbert_score, it re-runs the scorer on those papers before writing the site data. This means adding a new scorer to PF automatically backfills the entire historical database on the next run.

Pros
  • The site always shows complete, consistent score fields across all papers
  • Upgrading the scoring method does not require a separate migration script
Cons
  • Retro-scoring can be expensive if the backlog is large (re-encodes all missing papers through SBERT)
  • Makes publish time unpredictable — first run after a scorer upgrade is much slower

STEP 11 Loop Control & Automation

A one-shot pipeline is a tool. A loop is a system. Loop engineering determines how well the pipeline adapts between runs without requiring manual reconfiguration.

Dynamic-LR — loop_control.py

A dedicated state-machine module with documented deterministic rules. Reads from query_state and runs tables to produce per-run configuration adjustments: query priority ordering, dynamic filter thresholds, fallback behavior for target papers. Rules are enumerated and documented — no hidden logic.

PRS — LLM refinement loop

results_acceptable，LLM 会重写查询并继续运行。这是一个智能体式、非确定性的循环——而非可重复的自动化流水线。">The "loop" in PRS is an intra-run feedback loop driven by Ollama. After scoring, if results fail results_acceptable, the LLM rewrites the query and the run continues. This is an agentic, non-deterministic loop — not a repeatable automation pipeline.

PF — No loop control

PF is a single-pass pipeline. It has no inter-run state (besides the embedding cache) and no adaptive behavior. The same config produces the same queries every run. The only adaptive element is the date-gap validator which surfaces missing days — but does not act on them.

Deep Dive

Dynamic-LR Loop Control Rules

Query priority demotion

If a query produces zero accepted papers for N consecutive runs, it is assigned lower priority and executed later (or skipped if max_queries is tight). Queries with consistently high acceptance rates are promoted to run first.

Why
  • API calls are not free — stop wasting budget on queries that produce nothing
  • Keeps the run within rate-limit budget as the database matures

High duplicate-rate response

If duplicate_rate > 0.80 in recent runs, the loop control adds a recency filter (publication date after last run) to prefer newly published papers over ones already in the database.

Why
  • As the database grows, more retrieved papers are already known
  • Without adaptation, the pipeline wastes API calls re-retrieving old papers

PF — Date Gap Detection

# Check which dates in the expected window have zero papers
gaps = [d.isoformat() for d in date_range
        if date_counts[d.isoformat()] == 0]

This is the only adaptive quality check in PF — it detects days in the days_back window that have no papers, suggesting the fetch may have missed entries.

Pros
  • Visible signal for operability monitoring — gaps are printed and included in the run summary
  • Distinguishes "no papers published that day" from "papers were missed"
Cons
  • Detection only — no automatic retry or fallback action
  • Many topics will genuinely have zero papers on weekends; all gaps look the same
  • Not applicable to year-window baselines (Dynamic-LR, PRS) which don't use a rolling date window

Target Paper Detection (Dynamic-LR only)

If target_papers are configured, Dynamic-LR uses a five-level fallback chain to check whether each target was found and accepted:

1. DOI lookup via S2 paper/{doi}
2. S2 paperId lookup via S2 paper/{id}
3. arXiv ID lookup
4. Exact normalized title match in database
5. Quoted title search against S2 API
Pros
  • Directly measures retrieval recall on known ground-truth papers
  • Identifies whether a missed paper was not found or was found but rejected (and why)
  • Enables structured evaluation of baseline vs. agent architectures
Cons
  • Requires manual curation of target papers — effort for the user
  • Fallback lookups add API calls not counted in query metrics

Synthesis — Building the Ultimate Baseline

Each of the ten recommendations below borrows a proven element from one system and integrates it into Dynamic-LR's architecture. They are ordered by expected impact on retrieval quality and coverage.

1

Add SBERT + FAISS Scoring

Adopt PF's score_papers_sbert + faiss.IndexFlatIP as the primary scorer. Keep Dynamic-LR's lexical formula as the explainability companion. Store both scores on every paper — the pattern PF already established. The SBERT score drives the accept/reject decision; the lexical score drives the reject evidence and audit trail. Add PF's numpy embedding cache to avoid re-encoding on repeat runs.

2

Add arXiv Source Adapter

Add a modular arXiv fetcher following the interface used by semantic_scholar_client.py. Use PRS's XML parser for reliability. Route results through the existing normalizer.py and identity.py so arXiv papers enter the same deduplication chain as S2 papers.

3

Add OpenAlex Source Adapter

Add OpenAlex following PF's fetch_openalex pattern, including PF's abstract reconstruction from inverted index. Add OpenAlex ID as a fifth identity type in identity.py. OpenAlex is free and keyless — it adds coverage without any quota pressure on S2.

4

Upgrade to Async Concurrent Fetch

Wrap all three source fetchers in an asyncio.gather call following PF's pattern. Each source runs in parallel; failures return exceptions without aborting the others. Estimated speedup: 2–3× on multi-source runs.

5

Upgrade S2 Client with PRS Backoff

Replace the current 429 handler in semantic_scholar_client.py with PRS's pattern: read Retry-After header first; fall back to exponential backoff with jitter (base × 2^attempt × uniform(1.0, 1.5)). Cap at 60 s. This prevents synchronized retries from multiple parallel workers.

6

Adopt Completeness-Based Merge Selection

In deduper.py, add PF's _completeness_score before the merge step. If the incoming candidate scores higher than the existing record, make the incoming candidate the base and fill its missing fields from the existing record. This ensures the richest available record is always canonical.

7

Persist SBERT Embedding Cache

Add PF's cache.py pattern: save a numpy array of embeddings keyed by paper_id, with a metadata file storing the model name for auto-reset on model change. On repeat runs, only newly retrieved papers need encoding. As the database grows this becomes critical for run time.

8

Add Date-Gap Validation

Add PF's validate_fetch_results as a post-fetch check. For year-window pipelines, adapt it to check coverage by year rather than day. Include the gap summary in the run artifact and ManagerRunResult.

9

Add ML Acronym Expansion

Add PRS's ML_ACRONYM_EXPANSIONS lookup table to query_builder.py as a static pre-processing step applied to the topic string before generating the combined query. This is deterministic, dependency-free, and recovers retrieval coverage on common abbreviations (rag, llm, rlhf, cot) without an LLM.

10

Deterministic Multi-Round Fallback in Loop Control

Replace the LLM refinement concept from PRS with a deterministic fallback rule in loop_control.py: if good_count < 3 after scoring, widen the query set on the next run by including the next-tier query hints. If duplicate_rate > 0.80, add a date filter. This captures the intent of PRS's refinement loop without any LLM dependency.

Final note on determinism: SBERT embeddings are deterministic — the same text through the same fixed-weight model produces the same vector every time. Adding SBERT does not violate the baseline's determinism requirement. It only violates the "no embeddings in v1" constraint, which was explicitly listed as a first-iteration limitation to be revisited. All ten recommendations above produce fully reproducible results for the same config, database state, and API responses.
